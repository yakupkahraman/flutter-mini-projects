# Contributing

Thanks for wanting to contribute! This repo is a learning resource, so the bar for
a good contribution isn't "clever code" — it's **code a beginner can read and learn
from**.

There are three ways to help:

- [Adding a new project](#adding-a-new-project)
- [Improving an existing project](#improving-an-existing-project)
- [Reporting a bug](#reporting-a-bug)

## Adding a New Project

### 1. Naming

Project folders live under `projects/` and **must** follow this pattern:

```
projects/0N_snake_case_name/
```

- **Two digits, zero-padded** — `08_todo_app`, not `8_todo_app`
- **Use the next free number.** The last project is `07_weather_app`, so a new one
  starts at `08`. Don't renumber or reorder existing projects — their folder names
  are linked from the root README and from other projects' docs.
- **`snake_case`, all lowercase** — `08_todo_app`, not `08_TodoApp` or `08_todo-app`
- **The folder name after the number must match the `name:` field in
  `pubspec.yaml`.** For example `projects/08_todo_app/` goes with
  `name: todo_app`. Every existing project follows this, and Dart's own package
  naming rules require `snake_case` here anyway.

Past `09`, just keep counting — `10_`, `11_`, and so on. The padding stays two
digits so folders keep sorting correctly.

### 2. What makes a good project here

Projects are roughly ordered by difficulty, and each one should **teach something
the earlier ones don't**. Before starting, check the table in the
[root README](README.md) — if your idea only covers concepts already demonstrated,
it probably belongs as an improvement to an existing project instead.

A good fit is:

- **Small.** One clear idea, a handful of screens. If it needs a week to read, it's
  too big.
- **Self-contained.** It runs on its own with `flutter pub get && flutter run`. No
  backend you have to set up, no paid services. A free API key is fine — document
  how to get one.
- **Readable over clever.** Prefer the obvious solution. If you do something
  non-obvious, explain why in the project README.

### 3. What to commit

Only the source and config are tracked — the generated platform folders are not:

```
projects/08_todo_app/
├── lib/                    ✅ commit
├── assets/                 ✅ commit (only if the app uses them)
├── pubspec.yaml            ✅ commit
├── pubspec.lock            ✅ commit
├── analysis_options.yaml   ✅ commit
├── README.md               ✅ commit
├── android/  ios/  web/    ❌ don't commit — generated by `flutter create .`
├── build/                  ❌ don't commit
└── .dart_tool/             ❌ don't commit
```

This keeps the repo small. Anyone cloning it regenerates the platform folders with
the `flutter create --platforms=... .` step documented in the root README.

> If your app needs platform-specific setup — permissions, an API key, a minimum
> SDK version — **don't commit the platform folder to work around it.** Write the
> steps in your project's README instead, the way
> [07_weather_app](projects/07_weather_app) documents its location permissions.

### 4. Write the project README

Every project has a README, and they follow the same shape so readers know where to
look. Copy the structure from an existing one —
[05_study_tracker](projects/05_study_tracker) is a good middle-of-the-road example:

```markdown
# Project Name

🟢 **Beginner** · One-line description.

A short paragraph on what the app actually does from the user's side.

## 📸 Screenshots

<p align="center">
  <img src="../../docs/08_todo_app/home.png" width="240" alt="Home screen" />
</p>

## What You'll Learn

- Bullet points, one concept each

## Project Structure

Tree of `lib/`, with a comment on what each file is for.

## Key Concepts

The interesting parts, with short code snippets **taken from the actual code**.

## Getting Started

Prerequisites, `flutter pub get`, the `flutter create` step, `flutter run`.

## Try It Yourself

3–4 exercises for the reader to extend the app.
```

Pick a difficulty badge that's honest:

| Badge | Level | Roughly means |
|-------|-------|---------------|
| 🟢 | Beginner | `setState`, layout, widgets — no packages beyond the basics |
| 🟡 | Intermediate | Local storage, charts, network calls, async |
| 🔴 | Advanced | State management packages, databases, code generation |

**Code snippets must match the code in `lib/`.** A snippet that drifted out of sync
is worse than no snippet — someone will copy it and wonder why it doesn't compile.

### 5. Screenshots

Screenshots go in `docs/` under a folder matching the project:

```
docs/08_todo_app/home.png
```

- Reference them from the project README with the relative path
  `../../docs/08_todo_app/home.png`
- Use `width="240"` so tall phone screenshots don't dominate the page
- Keep them reasonably sized — a few hundred KB each is plenty
- GIFs are welcome for anything animated (a game round, a swipe gesture)

### 6. Update the root README

Add one row to the table in [README.md](README.md), in number order:

```markdown
| 08 | [Todo App](projects/08_todo_app) | 🟢 Beginner | What it teaches, briefly |
```

Keep the "What it teaches" column concrete — name the actual widgets, packages or
APIs (`GridView.builder`, `shared_preferences`) rather than vague phrases like
"state management and layout".

If your project introduces a new package, add it to the **Tech Stack** line too.

## Improving an Existing Project

Very welcome, and often more useful than a new project. Good examples:

- Fixing a bug, a crash, or a deprecated API
- Clarifying a confusing README section, or adding a missing concept
- Adding a screenshot or GIF where there isn't one

One thing to keep in mind: **these apps are deliberately simple.** Refactoring
`01_bmi_calculator` to use Riverpod would make it worse for its purpose — a
beginner reading it should meet `setState` first. If a project's simplicity feels
like a limitation, that's usually a sign it should be demonstrated in a *later*
project instead.

If you change code that a README quotes in its "Key Concepts" section, update the
snippet in the same PR.

## Reporting a Bug

Open an issue with:

- Which project (`03_xox_game`)
- What you expected, and what happened instead
- Your platform and `flutter --version` output
- The error message or a screenshot, if there is one

Platform setup problems are the most common issue — if the app won't build at all,
double-check you ran the `flutter create --platforms=... .` step first.

## Before You Open a PR

Run these in your project folder:

```bash
dart format .
flutter analyze
```

`flutter analyze` should come back clean. Every project ships with
`analysis_options.yaml` using `flutter_lints`, so this is just the default rule set
— nothing exotic.

Then check:

- [ ] Folder is named `0N_snake_case_name`, using the next free number
- [ ] Folder name matches `name:` in `pubspec.yaml`
- [ ] No `android/`, `ios/`, `build/` or `.dart_tool/` in the diff
- [ ] Project README follows the structure above, with a difficulty badge
- [ ] Code snippets in the README match the real code
- [ ] Screenshots are in `docs/0N_project_name/`
- [ ] Root README table has a row for the project
- [ ] The app runs from a fresh clone with the documented steps

Keep PRs focused — one project or one fix per PR is much easier to review than a
batch of unrelated changes.

## Questions

Not sure whether your idea fits, or where a concept belongs? Open an issue and ask
before building it. That's usually faster than finding out in review.
